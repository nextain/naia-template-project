# Gemini 3.5 — LLM 역할분담/어댑터 설계 적대검토

적대적 검토자로서 이 설계의 치명적인 빈틈과 실패 모드를 냉정하게 지적한다.

### 1. 역할 분담 (우려)
**핑퐁 루프와 비용 폭탄.** 작은 모델이 오탐(False Positive)을 내면, 큰 모델은 존재하지 않는 오류를 검증하느라 토큰과 시간을 낭비한다. "반드시 cross-check"의 종착지(최종 결정권자)가 모호하여, 두 모델 간 의견 불일치 시 무한 루프에 빠지거나 정지할 위험이 크다.

### 2. 디텍트 계층 (우려)
**의미론적 왜곡(회색 지대) 방치.** 결정론적 스크립트는 구조만 보고, 작은 모델은 맥락이 부족하다. 구조는 완벽하지만 번역 과정에서 미묘하게 비즈니스 로직이 왜곡된 '중간 크기 오류'는 작은 모델이 지적하지 못하므로, 큰 모델에게 전달조차 되지 않고 최종 통과된다.

### 3. CLI 어댑터 (우려)
**극도의 취약성.** SDK가 아닌 로컬 CLI(`claude`, `codex`)를 `exec`로 호출하는 것은 시한폭탄이다. CLI 업데이트로 인한 출력 포맷 변경, ANSI 이스케이프 문자 혼입, 세션 만료, Rate Limit 발생 시 파싱 에러로 전체 파이프라인이 즉시 중단된다. CI/CD 환경에서 이식성이 제로에 가깝다.

### 4. 테스트 무결성 (우려)
**합격 연극(Theater of Security).** 소스 코드에 특정 문구가 있는지 검사하는 grep 방식 테스트는 실제 런타임 동작을 보장하지 않는다. CLI 호출 실패, 네트워크 타임아웃, 모델 거부 반응(예: codex의 gpt-5-mini 거부) 시의 예외 처리 흐름을 검증하는 Mock 테스트가 전무하다.

### 5. 표준화 가치 (우려)
**표준 박제 불가.** 로컬 CLI 바이너리에 의존하는 설계는 환경 전파가 불가능하여 곧 폐기될 운명이다.

---

### 🚨 가장 큰 리스크 (Single Biggest Risk)
**"CLI 래핑 의존성으로 인한 런타임 파이프라인의 무조건적 붕괴"**
인증 만료, CLI 도구의 마이너 업데이트(출력 형식 변경), 혹은 CI 환경의 바이너리 부재 중 단 하나만 발생해도 동기화 스크립트는 아무런 예외 처리 없이 크래시된다. SDK 기반 호출로 즉시 전환해야 한다.
