# 배운 교훈 (Lessons Learned)

> 이 문서는 naia 생태계 프로젝트를 운영하면서 실제로 부딪힌 문제와 해결책을 기록한다.
> 규칙이 아니라 **왜** 그 규칙이 존재하는지를 설명한다.
> 새 프로젝트 시작 시, 새 팀원(AI 포함) 온보딩 시 반드시 읽는다.

---

## L01 — 프로세스 없이 코딩 먼저 하면 반드시 다시 한다

**무슨 일이 있었나**: AI가 "먼저 빠르게 해볼게요"라며 사용자 시나리오, 요구사항 없이 코드부터 작성했다.
나중에 방향이 틀렸음을 깨닫고 전부 다시 작성했다.

**해결책**: P01(UC) → P02(테스트시나리오) → P03(요구사항) 게이트 없이는 코드 한 줄도 금지.
코드 작성 속도보다 **방향의 정확성**이 훨씬 중요하다.

**규칙화**: `agents-rules.json` `required_process_gate` F04.

---

## L02 — 등록되지 않은 파일/폴더는 반드시 쌓인다

**무슨 일이 있었나**: AI가 `experiments/`, `vendor/`, `adapter-grpc/` 등을 허가 없이 루트에 만들었다.
시간이 지나면서 루트가 누가 만들었는지 모르는 파일들로 오염됐다.

**해결책**: F12/F13 화이트리스트. 미등록 리소스는 경고가 아니라 **삭제**.
"차라리 깨지는 게 낫다" — 등록되지 않은 것은 존재해선 안 된다.

**규칙화**: `agents-rules.json` F12/F13 + `enforce-root-structure.sh`.

---

## L03 — AI가 정책을 만들어내면 안 된다

**무슨 일이 있었나**: AI가 README에 "Breaking changes require 4-week advance notice"를 추가했다.
사용자가 한 번도 말한 적 없는 정책이었다.

**해결책**: 코드/문서에 추가하는 모든 정책·규칙은 사용자가 명시적으로 말한 것만.
"누가 이래? 나 그렇게 말한 적 없어"가 나오면 즉시 제거.

**규칙화**: Karpathy 원칙 2 (Simplicity First) — 요청된 것만.

---

## L04 — 세션이 끊기면 문맥이 사라진다

**무슨 일이 있었나**: 세션이 새로 시작될 때마다 AI는 이전 작업 상태를 모른다.
"지금 어디까지 했지?" 를 매번 다시 설명해야 했다.

**해결책**: `process-status.json` — 세션 시작 시 읽고 `last_updated` 갱신,
세션 종료 시 완료 게이트·deliverable 업데이트.
이 파일 하나로 어떤 세션이든 즉시 현황 파악 가능.

**규칙화**: `agents-rules.json` `mandatory_reads` + F05.

---

## L05 — 헌장은 AI가 수정하면 안 된다

**무슨 일이 있었나**: AI가 "더 나은 것 같아서" AGENTS.md 규칙을 슬며시 수정했다.
사용자가 한참 후에 발견하고 롤백했다.

**해결책**: 헌장 파일 (AGENTS.md, agents-rules.json, process-status.json, project-structure.md)은
사용자 명시 승인 없이 AI가 절대 수정 불가.

**규칙화**: `agents-rules.json` `charter_immutability` F01.

---

## L06 — 명칭 혼란은 오래 간다

**무슨 일이 있었나**: `alpha-memory`와 `naia-memory`가 여러 파일에 혼용됐다.
어떤 문서는 alpha-memory, 어떤 문서는 naia-memory — 몇 달 동안 두 이름이 공존했다.

**해결책**: 용어사전(`docs/glossary.md`)을 SoT로 유지.
코드·문서·이슈에서 용어를 쓸 때 glossary 항목 먼저 확인.

**규칙화**: `docs/glossary.md` 신설 + 커밋 전 명칭 일관성 확인.

---

## L07 — 역할 경계가 흐려지면 코드가 잘못된 곳에 들어간다

**무슨 일이 있었나**: naia-agent에 naia-os 전용 기능(HTTP 서버, 오디오 등)이 들어갔다.
경계를 누가 결정하는지 불명확해서 모두 naia-agent에 밀어넣었다.

**해결책**: 경계 판단 기준 2가지 모두 YES여야 naia-agent에 속함:
1. "CLI로 단독 실행할 때 의미있는가?"
2. "LLM이 관리하고 호출하는가?"

**규칙화**: `AGENTS.md` 경계 원칙 섹션.

---

## L08 — 빠른 이동보다 올바른 이동이 낫다

**무슨 일이 있었나**: "일단 빠르게 만들고 나중에 정리하자"는 접근이 기술 부채를 쌓았다.
나중에 정리하는 일은 대부분 처음부터 다시 만드는 것보다 더 오래 걸렸다.

**해결책**: 프로세스를 지키는 데 드는 시간 < 잘못된 방향으로 간 후 다시 하는 시간.
"구조를 만드는 게 먼저."

**규칙화**: 이 템플릿 자체가 교훈의 결과물.

---

## L09 — 도구(harness)가 선결이다

**무슨 일이 있었나**: 코드부터 짜기 시작했다가 나중에 "어, AI가 규칙을 모르네"를 발견했다.
도구(AGENTS.md, agents-rules.json, enforce 스크립트)가 없으면 규칙은 말뿐이다.

**해결책**: 새 프로젝트 시작 시 이 template-project를 복사해서 시작.
코드보다 harness가 먼저.

**규칙화**: 이 템플릿의 존재 이유.

---

## Karpathy 4원칙 (항상 적용)

1. **Think Before Coding** — 가정을 명시적으로 밝힌다. 불명확하면 질문한다.
2. **Simplicity First** — 요청된 최소한의 코드만. 추가 기능·에러 핸들링·추상화 금지.
3. **Surgical Changes** — 변경해야 할 것만. 인접 코드 "개선" 금지.
4. **Goal-Driven Execution** — 성공 기준 정의 → 검증될 때까지 반복.
